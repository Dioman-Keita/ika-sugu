# Admin Product Authoring Specification

## Branch

- Working branch: `feature/admin-product-authoring`

## Goal

Build a real admin product authoring workflow, not a minimal product form.

The current admin UI is too limited compared to the product model already supported by the backend and Prisma schema. The target is a proper back-office flow for creating and editing products, variants, translations, images, pricing, and publication state.

This specification captures:

- current gaps
- product decisions already made
- required fields and validations
- recommended Prisma/backend upgrades
- implementation priorities

## Product decisions locked in

### SKU generation

- SKU must be generated by the backend.
- SKU must not be entered manually by the admin user.
- The UI may display generated SKUs after creation or in edit mode, but it should not require manual entry.
- SKU generation must produce stable, unique values.

Recommended format options:

- product-slug based with normalized color/size suffix
- or opaque generated SKU with predictable prefix

Minimum requirement:

- global uniqueness in `ProductVariant.sku`

### Supported translations

- Supported locales for product authoring are currently:
  - French (`fr`)
  - English (`en`)

- The admin must provide both languages.
- If the initial/base content is entered in French, the admin must also enter the English translation.
- If the initial/base content is entered in English, the admin must also enter the French translation.

Minimum translated fields:

- product name
- product description

Recommended translated structured fields:

- specs.material
- specs.care
- specs.fit
- specs.pattern

### Select inputs

Anything that can be a dropdown should be a dropdown.

This applies at minimum to:

- `dressStyle`
- category
- currency
- publication status
- locale selector
- variant size
- variant color when driven by a controlled catalog
- yes/no flags where appropriate

Free text should be kept only where the value is genuinely open-ended.

### Back-office scope

The target is a complete back-office implementation, even if Prisma updates are required.

That means we should not stop at patching the existing form. We should close the gap between:

- what the UI can author
- what the backend expects
- what the product domain actually needs

## Current observed problems

### Admin product form is underpowered

The current form supports only a narrow subset of the product model:

- name
- slug
- description
- category
- base price
- discount percentage
- VAT
- dress style as free text
- basic variant fields

Missing or insufficient areas:

- no translation authoring
- no structured specs authoring
- no variant active/inactive toggle
- no publication workflow
- no backend-generated SKU flow
- no controlled dropdowns for several categorical fields
- no rich validation feedback
- no inventory/business constraints shown to the user

### Image upload is a likely failure point

The current image uploader uploads directly from the client to Supabase storage and depends on:

- public Supabase client env vars
- bucket existence
- bucket policies
- allowed remote image rendering
- correct public URL generation

Potential failure sources:

- missing `NEXT_PUBLIC_SUPABASE_URL`
- missing publishable key
- bucket `products` missing
- storage policy denies upload
- unsupported image domain in Next image config
- upload succeeds but UI state is not persisted as expected
- no server-side cleanup for deleted images

### UI/backend contract mismatch

The backend and schema already support more than the form exposes:

- `ProductTranslation`
- `specs` JSON
- `ProductVariant.isActive`
- variant-level pricing and compare-at pricing
- richer variant identity

The authoring UI must be upgraded to match the actual domain.

## Required product authoring contract

### Product core fields

Required:

- base locale
- `name`
- `slug`
- `description`
- `categoryId`
- `dressStyle`
- `basePrice`
- `discountPercentage`
- `vatRate`
- publication status

Recommended:

- internal admin note
- featured flag
- merchandising order / priority
- tags

### Product translations

For both `fr` and `en`:

- translated name
- translated description

Recommended:

- translated specs:
  - material
  - care
  - fit
  - pattern

Validation rules:

- both locales are required
- the two locale entries must not be empty
- base locale and secondary locale must be explicit in the UI

### Variant fields

For each variant:

- backend-generated SKU
- color
- color hex
- size
- price
- compare-at price
- currency
- stock
- active/inactive status
- image gallery
- cover image

Validation rules:

- `colorName` required
- `size` required
- `price >= 0`
- `stock >= 0`
- `compareAtPrice >= price` when present
- unique variant identity per product
- at least one usable image strategy must exist

### Image requirements

Per variant:

- multiple images
- one cover image
- drag/drop upload
- upload progress
- failure state
- retry flow
- remove image
- reorder images

Recommended:

- server-side delete of removed assets
- image alt text if later needed

## Dropdown-first design rules

### Must be dropdowns

- `dressStyle`
  - recommended values:
    - `casual`
    - `formal`
    - `party`
    - `gym`

- category
  - loaded from DB

- currency
  - initial supported list:
    - `USD`
    - `EUR`
    - `XOF`

- publication status
  - if Prisma is updated:
    - `DRAFT`
    - `PUBLISHED`
    - `ARCHIVED`

- base locale
  - `fr`
  - `en`

- size
  - initial supported list:
    - `XX-Small`
    - `X-Small`
    - `Small`
    - `Medium`
    - `Large`
    - `X-Large`
    - `XX-Large`
    - `3X-Large`
    - `4X-Large`
    - `Unique`

### Good candidates for dropdowns

- color
  - if business wants a controlled catalog instead of free text
- VAT rate
  - if rates are standardized
- product visibility flags

## Recommended Prisma upgrades

These upgrades are justified if the objective is a real back office.

### Product publication state

Add a product status field.

Recommended:

- `status ProductStatus @default(DRAFT)`

Enum:

- `DRAFT`
- `PUBLISHED`
- `ARCHIVED`

### Variant status control

Already exists:

- `ProductVariant.isActive`

UI and backend flows must fully support it.

### Base locale on product

Recommended:

- store the source locale used for original authoring

Possible field:

- `sourceLocale String @default("fr")`

### Optional merchandising fields

Recommended if catalog management matters:

- `isFeatured Boolean @default(false)`
- `sortOrder Int @default(0)`
- `tags String[]`

### Optional audit metadata

Recommended for admin operations:

- `createdById`
- `updatedById`

## Recommended backend behavior

### SKU generation rules

The backend should generate SKU values during create and update.

Requirements:

- deterministic enough to inspect
- unique
- regenerated only when needed
- stable on edits that do not affect variant identity

Suggested identity inputs:

- product slug
- color slug
- size slug
- short suffix for collision handling

### Validation

Server-side validation must reject:

- empty translated names/descriptions
- duplicate slug
- duplicate variant identity
- invalid price combinations
- negative stock
- unknown category
- invalid locale set
- unsupported dropdown values

### Image handling

Recommended direction:

- move critical storage operations behind server-side endpoints/actions where possible
- support asset deletion when an image is removed from a variant
- prevent orphaned storage files

### Product write model

Create/update operations should support:

- product core data
- translations
- variants
- variant images
- variant activation state
- publication status

within a coherent transactional workflow.

- add variant `isActive` support

## Non-negotiable implementation outcomes

- SKU is backend-generated
- FR and EN translations are both authored in admin
- categorical fields use dropdowns whenever practical
- admin product authoring covers the real product domain
- implementation may update Prisma if needed
